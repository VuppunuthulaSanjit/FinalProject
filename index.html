What I’m trying to make / solve

Most VR driving demos look cool but feel fake: the visuals move, but your body doesn’t. The goal is to add physical cues (motion + steering feedback) so the driving experience feels more real.

Project goal

  • Immersion: connect hardware motion cues to driving actions.
  • Responsiveness: motion reacts in real time (low latency, repeatable behavior).
  • Buildability: prototype under school/time constraints while documenting clearly.

Success criteria

  • A working prototype that proves the control loop (command → Arduino → motion output).
  • Proof-first documentation: wiring, code, timeline, blockers, and testing notes.
  • Clear next steps showing how the full-scale version becomes achievable later.

What I built

Two-phase approach: design/research first, then a mini prototype pivot to “ship” a working build and prove the most important technical requirement.

Phase 1 (Weeks 1–9): full rig concept

Seat motion plan + actuator research + architecture for future VR integration. This phase produced the “what we want to build” and the engineering plan to get there.

Key lesson: “fast” actuator specs are often no-load numbers. Under real load, speed drops. Testing under load matters.

Phase 2 (Weeks 10–13): mini prototype pivot

A smaller build demonstrating responsive motion/steering control on command. This proves the control loop, while mechanical scale-up becomes the next milestone.

Why the pivot is defendable: it keeps progress real. The project ends with a working prototype + documented blockers, not just a big plan.

Proof: Wiring + Arduino control (linear actuator example)

Step-by-step wiring and a serial-command Arduino sketch (EXTEND / RETRACT / STOP) for actuator control. If you didn’t end up using the actuator in the final mini build, keep this as documented research/proof-of-learning. 

// Basic serial-controlled linear actuator (example)
int in1 = 8;
int in2 = 9;
int ena = 10;

void setup() {
  pinMode(in1, OUTPUT);
  pinMode(in2, OUTPUT);
  pinMode(ena, OUTPUT);
  Serial.begin(9600);
}

void loop() {
  if (Serial.available()) {
    String cmd = Serial.readStringUntil('\n');
    cmd.trim();

    if (cmd == "EXTEND") {
      digitalWrite(in1, HIGH);
      digitalWrite(in2, LOW);
      analogWrite(ena, 255);
    } else if (cmd == "RETRACT") {
      digitalWrite(in1, LOW);
      digitalWrite(in2, HIGH);
      analogWrite(ena, 255);
    } else if (cmd == "STOP") {
      digitalWrite(in1, LOW);
      digitalWrite(in2, LOW);
    }
  }
}

Proof: Mini prototype (servo/motor motion)

This is the delivered build: a mini version proving command → Arduino → motion output. The point is low-latency, repeatable control (not perfect realism yet). 

#include <Servo.h>
Servo s;
const int servoPin = 6;

void setup() {
  s.attach(servoPin);
  s.write(90); // center
  Serial.begin(9600);
}

void loop() {
  if (Serial.available()) {
    String cmd = Serial.readStringUntil('\n');
    cmd.trim();

    if (cmd == "LEFT")   s.write(45);
    if (cmd == "RIGHT")  s.write(135);
    if (cmd == "CENTER") s.write(90);
  }
}
Why this matters: scaling up is “just engineering work” once the control loop is proven. The mini build proves the hardest part under real school constraints.

Next steps

These are testable, measurable actions that move from “working prototype” to “wheel-driven motion system.”

Immediate (next 1–2 weeks)

  • Implement a PC-side script that streams wheel values smoothly (rate limit + smoothing).
  • Define mapping from wheel angle → servo angle (calibration + deadzone).
  • Measure latency + jitter (what “real-time” is in milliseconds).

Near-term milestone

  • Add failsafes: software STOP + physical kill switch for any motorized motion.
  • Upgrade driver hardware if overheating/instability appears under load.
  • Re-mount mechanism to reduce flex so motion is repeatable and safe.

Level-up milestone

  • Replace “command-only” with real input integration (wheel telemetry → live motion).
  • Explore adding VR + motion cueing logic (comfort tuning, soft start/stop curves).
  • Document the final architecture as a reproducible build guide.
