Mission

Most VR driving demos look impressive but don’t feel physical: the visuals move, but your body gets no motion cue. Our goal was to build a VR racing simulator that adds real hardware motion/steering response so driving actions feel more believable.

What I’m trying to solve

  • Immersion gap: VR visuals alone feel “floaty” without physical cues.
  • Responsiveness: motion must react fast enough to feel linked to inputs.
  • Build reality: prove the control loop first, then scale the mechanics.

Success criteria

  • A working prototype where commands reliably create motion output.
  • Clear evidence: photos, links, and reproducible code.
  • A documented roadmap from “mini build” to “full rig.”

Build overview

We treated the semester like a real engineering program: start with research + architecture, then pivot when constraints are discovered, then ship a prototype that proves the core loop and sets up the next iteration.

Phase 1 (Weeks 1–9): full rig concept

We planned a seat-motion VR simulator and researched actuators, motion types, control wiring, and the PC→Arduino command pipeline. This phase produced the “how it should work” blueprint, plus early code/wiring proof for actuator control.

Key lesson: “fast” actuator listings often describe no-load speed. Under real load, motion becomes slower and less realistic. That constraint pushed us toward a pivot that kept progress measurable.

Phase 2 (Weeks 10–13): mini prototype pivot

We built a smaller model using available parts to prove the real-time control loop and deliver a working demonstration. This let us finish with a functional system and a clear, evidence-backed plan for scaling up.

Deliverable mindset: a prototype that works + honest blockers + next steps is stronger than a big concept that never runs.

Code: WASD control for an actuator (PC → Arduino)

This is a clean, presentation-ready example of controlling an actuator using W/S/X from a PC keyboard. The PC sends single-character commands over Serial; Arduino drives an H-bridge. This mirrors the same command-loop you proved in the mini build.

Hardware note: For real actuators under load, use a proper motor driver (ex: BTS7960 / IBT-2 or a rated DC motor controller). L298N is often too weak for continuous high-current actuator use. If you’re only demoing small loads, it may survive, but it’s not ideal.

Arduino (receives W/S/X over Serial)

W = extend/up, S = retract/down, X = stop. Uses two direction pins + one PWM pin (ENA). 

// Actuator control via Serial keyboard commands
// W = extend, S = retract, X = stop

const int IN1 = 8;
const int IN2 = 9;
const int ENA = 10;     // PWM speed (0-255)
int speedVal = 220;     // tune for smoother motion

void setup() {
  pinMode(IN1, OUTPUT);
  pinMode(IN2, OUTPUT);
  pinMode(ENA, OUTPUT);

  stopActuator();
  Serial.begin(115200);
}

void loop() {
  if (!Serial.available()) return;

  char c = Serial.read();
  if (c == 'W' || c == 'w') extendActuator();
  else if (c == 'S' || c == 's') retractActuator();
  else if (c == 'X' || c == 'x') stopActuator();
}

void extendActuator() {
  digitalWrite(IN1, HIGH);
  digitalWrite(IN2, LOW);
  analogWrite(ENA, speedVal);
}

void retractActuator() {
  digitalWrite(IN1, LOW);
  digitalWrite(IN2, HIGH);
  analogWrite(ENA, speedVal);
}

void stopActuator() {
  digitalWrite(IN1, LOW);
  digitalWrite(IN2, LOW);
  analogWrite(ENA, 0);
}

PC Python (sends WASD over Serial)

Uses pynput to read keys. Replace COM3 with your port. 

# pip install pyserial pynput
import serial
from pynput import keyboard

PORT = "COM3"      # Windows example. On Mac/Linux: "/dev/tty.usbmodemXXXX"
BAUD = 115200

ser = serial.Serial(PORT, BAUD, timeout=0.1)

print("W = extend, S = retract, X = stop, ESC = quit")

def send(ch: str):
  try:
    ser.write(ch.encode("utf-8"))
  except Exception as e:
    print("Serial write error:", e)

def on_press(key):
  try:
    k = key.char
    if k in ("w","W"): send("W")
    elif k in ("s","S"): send("S")
    elif k in ("x","X"): send("X")
  except AttributeError:
    # Non-char keys (arrows, etc.)
    pass

def on_release(key):
  if key == keyboard.Key.esc:
    send("X")
    return False

with keyboard.Listener(on_press=on_press, on_release=on_release) as listener:
  listener.join()

ser.close()
Why this belongs in your portfolio: It clearly demonstrates the control pipeline (PC input → Serial → Arduino → driver → motion), which is exactly what the full VR rig needs—just with bigger mechanics and safer hardware.

Timeline (Weeks 1–13)

Each entry is written like a real engineering log: what changed, what we learned, and what it enabled next.

1
Week 1 — Research plan + system vision Phase 1

We defined a clear end goal: a VR racing simulator that feels physical, not just visual. We outlined what “realism” should mean (fast response, believable cues) and mapped early constraints like cost, time, and build complexity. This set the foundation for a plan we could defend even if the hardware choices changed later.

2
Week 2 — Requirements + parts planning Phase 1

We broke the project into requirements: hardware motion, control electronics, and a pathway to VR integration. We also started a parts and tooling list and compared what was realistic to acquire within school limitations. The result was a more grounded scope, with a plan to prove the control loop early.

3
Week 3 — Technical approach + modular architecture Phase 1

We designed the system as modules instead of a single “giant build”: inputs, a PC command layer, serial communication, Arduino control, and the final actuator/motor output. We planned documentation like a real repo: clear commits, logs, and a reproducible wiring/code structure. This made later iteration easier because swapping hardware didn’t require rewriting the entire project story.

4
Week 4 — Motion realism research Phase 1

We compared different types of motion cues (heave vs pitch/roll) and learned that perceived realism often comes from timing and acceleration—not huge travel distance. We also realized that “fast enough” depends on how quickly motion reacts to input changes, especially in VR where mismatch can feel wrong. This week pushed us toward prioritizing responsiveness over maximum movement range.

5
Week 5 — Plan refinement + visualizing the goal Phase 1

We outlined an updated plan with clearer milestones and a tighter definition of what would count as success at the end of the semester. We used AI-assisted visualization to imagine the final system’s look and layout, which helped us reason about mounting, motion range, and user experience. We also researched viable parts and compared actuator, motor, and control options with a focus on realism and feasibility.

6
Week 6 — Timeline revision + reality check Phase 1

We revised the timeline to reflect what could actually be built and tested within the semester while still moving toward the VR rig goal. This week emphasized verification: identifying what needed testing under load and what assumptions were still unproven. It also clarified that hardware responsiveness would be the main risk factor, not “making it look cool.”

7
Week 7 — PC ↔ Arduino integration design Phase 1

We designed the command pipeline: the PC reads inputs, sends serial commands, and the Arduino converts those commands into motor/actuator behavior. We identified latency and jitter as the biggest threats to “feels real” motion, especially for VR comfort. This week turned the project from a mechanical idea into a true control-systems problem.

8
Week 8 — Wiring documentation + actuator control proof Phase 1

We produced step-by-step wiring documentation and wrote a serial-command control sketch (extend/retract/stop) to prove that the basic loop works. We also noted safety concerns like proper power delivery and driver capability, because actuators can pull high current under load. This week created “shareable proof” that we could actually control hardware—an important checkpoint before building bigger.

9
Week 9 — Feasibility check + speed constraint discovered Transition

We discovered the core blocker for the full rig: motion speed was not realistic enough using the initial actuator approach. Instead of pretending it was “fine,” we treated it like a real engineering constraint and began evaluating faster alternatives. This week is what triggered the pivot—because a working prototype mattered more than an oversized plan.

10
Week 10 — Pivot to a mini version (de-risk the project) Phase 2

We decided to build a mini version because the full-scale rig wasn’t viable under the time, cost, and speed constraints we confirmed. We researched tutorials and parts specifically chosen to prove responsiveness and control reliability first. This week reframed the goal as “ship a working control prototype,” while keeping the full rig as a planned expansion.

11
Week 11 — Attempt #1 scrapped (missing H-bridge) Phase 2

Jaydon and Sanjit found a build tutorial that looked promising, but it depended on an H-bridge driver we couldn’t access or acquire. Rather than forcing a sketchy substitute that could break the demo, we scrapped the approach and searched for a tutorial that matched available parts. This week reinforced a key habit: don’t build yourself into an un-debuggable corner.

12
Week 12 — Successful build + debugging to stability Phase 2

We found a simpler tutorial that used parts we actually had, and Jaydon and Sanjit built the hardware to completion near the end of the week. Mohith provided starter code, and the group debugged issues until behavior became stable and repeatable. This week is where the project “became real,” because the control loop worked reliably enough to demo.

13
Week 13 — Finalize + integration research (VR + wheel) Phase 2

We finalized the project and pushed beyond the basic demo by exploring VR integration and wheel-driven motion control. Evan Chen worked on the VR angle, while Sanjit researched connecting Logitech G27 wheel input so motion could respond directly to wheel commands. School Chromebook restrictions blocked serial/COM access, so we documented the limitation and found a workaround path by identifying tools/websites that can output G27 values for future forwarding on an unrestricted device.

Timeline narrative:

Big vision → real constraints discovered → smart pivot → working prototype shipped → integration research documented for the next iteration.

Next steps

These are concrete milestones that take the project from “command-based demo” to “wheel-driven motion system.”

Immediate

  • Run the WASD Serial pipeline on an unrestricted device and verify stable command timing.
  • Add smoothing / rate limiting so motion feels continuous instead of “steppy.”
  • Calibrate a mapping from wheel angle → actuator/servo position (deadzone + limits).

Scale-up milestone

  • Upgrade to a driver rated for the actuator’s current under load and add a physical kill-switch.
  • Rebuild the mechanics with rigidity (reduce flex) so motion is repeatable and safe.
  • Integrate VR comfort tuning: soft start/stop curves and motion cue timing.
